## 1. Что это за проект

**SWAIP** — российская PWA-социальная сеть. Пользователь получает мастер-ключ (7 слов + 3 цифры), из которого генерируется Ed25519 keypair. Публичный ключ (SHA-256 хэш) — это ID аккаунта.

Гость открывает профиль по инвайт-ссылке, видит голосовое приветствие, тематические посты. Зарегистрированный пользователь видит весь функционал.

**Продакшн URL:** `https://system-swaip.replit.app`

## 4. Что НЕЛЬЗЯ трогать никогда

### 🔴 App.tsx — структура файла

`App.tsx` — это **28 000+ строк** единого файла. Это архитектурное решение проекта.

**ЗАПРЕЩЕНО:**
- Делать полную перезапись файла (приведёт к потере всего)
- Добавлять второй `import React` (вызовет чёрный/белый экран)
- Дублировать import-строки
- Менять порядок объявлений `gActiveTheme` и `accent` (строки ~23289–23294) — TDZ-баг
- Добавлять `controllerchange` в `public/sw.js`
- Добавлять `|| caches.match('/')` как фолбэк для JS-файлов в service worker

**КАК ПРАВИЛЬНО редактировать App.tsx:**
```
Всегда использовать точечное редактирование (edit tool):
1. Скопировать 5-10 строк контекста вокруг нужного места
2. Указать ТОЛЬКО эти строки и изменения
3. Никогда не писать весь файл заново
```

### 🔴 CompassScreen

CompassScreen — главный экран навигации. **Не трогать без явного запроса пользователя.** Любые изменения в нём затрагивают всю навигацию приложения.

### 🔴 Service Worker (`artifacts/swaip/public/sw.js`)

Текущая версия: **v3**. Критичные правила:
- НЕ добавлять `controllerchange` в `index.html`
- НЕ добавлять `|| caches.match('/')` как фолбэк для JS-файлов
- При обновлении SW — тестировать на реальном устройстве перед деплоем

### 🔴 Система аутентификации

- Алгоритм: Ed25519. Не менять на другой.
- `getSessionToken()` в `sessionAuth.ts` — сначала читает cookie, потом заголовок. Порядок не менять.
- httpOnly cookie `swaip_token` — основной механизм. Не удалять.
- Таблица `sessions` — первичный ключ только `token`. Не менять тип.

### 🔴 Хранилище медиа

- Голосовые приветствия (`/api/greeting-upload`) → **GCS** (Google Cloud Storage)
- Аудио-посты (`/api/audio-upload`) → **локальный диск**
- Bucket: `replit-objstore-eeb01fc0-0643-4709-8f6f-5b134e4b5da8`
- **Не переносить аудио с локального диска в GCS** — поломает существующие ссылки

### 🔴 Платежи

**Stripe запрещён.** Только:
- ЮKassa
- CloudPayments
- Тинькофф

### 🔴 Первичные ключи в БД

Никогда не менять тип primary key столбцов (serial ↔ varchar). Ломает все данные.

Можно добавлять новые колонки или таблицы. После изменений:
```bash
pnpm --filter @workspace/db run push
```

**НЕЛЬЗЯ**: менять тип существующих ID-колонок.

### 🟡 GuestProfileScreen

Компонент показа гостевого профиля. При изменениях проверять:
- `isAuthenticated` — скрывает панель регистрации для залогиненных
- `isLoggedIn` — показывает фичи залогиненного (чат, подписки)
- `sessionChecking` — скрывает панель ПОКА идёт проверка cookie

### 🟡 Переводы (i18n)

Языковые строки разбросаны по `App.tsx` в объектах типа `const t = lang === 'ru' ? {...} : {...}`. Менять аккуратно, не нарушая структуру объектов.

## 9. Аутентификация и сессии

### Как работает

1. Пользователь вводит мастер-ключ (7 слов + 3 цифры)
2. Из ключа генерируется Ed25519 keypair (SHA-256 seed)
3. SHA-256 публичного ключа = `hash` (ID пользователя)
4. При входе создаётся сессия → сервер выдаёт токен
5. Токен хранится в httpOnly cookie `swaip_token` + в памяти (`_st`)

### `getSessionToken()` — порядок чтения

```typescript
// 1. Сначала cookie (httpOnly, безопасно)
const cookie = req.cookies?.['swaip_token'];
if (cookie) return cookie;

// 2. Потом заголовок (для старых клиентов)
return req.headers['x-session-token'] ?? '';
```

### Проверка сессии на клиенте (GuestProfileScreen)

```typescript
// isAuthenticated — залогинен ли вообще
const isAuthenticated = !!myHash && (!!getST() || cookieVerified);

// isLoggedIn — залогинен И смотрит ЧУЖОЙ профиль
const isLoggedIn = isAuthenticated && myHash !== hash;
```

- `sessionChecking = true` → скрываем панель регистрации пока идёт проверка cookie
- После `GET /api/session/me` → `cookieVerified = true`

## 13. Частые ошибки и их решения

### Чёрный/белый экран после правки App.tsx

**Причина:** Дублирующийся `import React` или нарушена структура JSX.  
**Решение:** Откатить изменение через Replit Checkpoint.

### Голосовое приветствие не играет

**Причина:** Браузер блокирует автоплей без жеста пользователя.  
**Норма:** При первом открытии может требоваться тап. Это ожидаемое поведение.

### Зарегистрированный пользователь видит панель регистрации

**Причина:** `isAuthenticated` не определён или `cookieVerified` не установился.  
**Решение:** Проверить, что `GET /api/session/me` возвращает `{ ok: true, hash }` при наличии cookie.

### Посты не показываются в гостевом профиле

**Причина:** Фильтрация по `authorMode` в `GET /api/broadcasts?author=hash&authorMode=pro`.  
**Решение:** Убедиться, что посты созданы в правильном режиме (pro/scene/krug).

### После обновления схемы БД всё сломалось

**Причина:** Изменён тип primary key.  
**Решение:** Откат изменений, никогда не менять тип ID-колонок.

### Инвайт-код CEO сбросился

**Причина:** `PUT /api/account` перезаписал `pro_customInviteCode`.  
**Это исправлено** — сервер теперь сохраняет кастомный код. Если сбросился — см. `CHANGE_INVITE_CODE.md`.
